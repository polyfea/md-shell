# Localization System

`polyfea-md-shell` includes a dynamic, decentralized localization system built on top of `@lit/localize`. It is designed for micro-frontend architectures where individual modules (web components) load their translations only when needed, while the Shell orchestrates the current active locale and applies global overrides.

## Key Features

1.  **Dynamic Loading**: Translation chunks are loaded lazily. If a user switches to French, only the French resources for currently registered modules are fetched.
2.  **Decentralized Registry**: Each functional module/component registers its own localization loader. The Shell does not need to know about all translations at build time.
3.  **Runtime Overrides**: Translations can be patched via external files (served from static pages or by dedicated microfrontend registered 
  in [Polyfea Controller](https://polyfea.github.io/documentation/)) without rebuilding the application code.

---

## 1. Locale Selection Logic

The active locale is resolved in the following priority:
1.  **User Selection**: Explicitly set via the `setLocale` method or the UI menu in the Shell.
2.  **Browser Preference**: If no user selection exists, the system checks `navigator.languages`.
3.  **Fallback**: If no match is found, the system displays the original strings defined in the source code (typically English).

### Resolution Algorithm
When loading a resource for a specific module, the system uses a **best-match strategy**:
1.  **Exact Match**: Tries to load `en-GB`.
2.  **Base Language**: If `en-GB` is not found, tries `en`.
3.  **Default**: If neither exists in the module's registration list, it keeps the source text.

## 2. Registering Dynamic Translations

Third-party components or features should register themselves using `registerLocaleModule`. This is usually done in the `connectedCallback` of the component.

<Source code={`
import { registerLocaleModule } from '@polyfea/polyfea-md-shell';

class MyFeature extends HTMLElement {
  connectedCallback() {
    super.connectedCallback();

    // Register translation loader for this module
    registerLocaleModule(
      '@my-org/auth-module',      // Unique module namespace
      ['en', 'sk', 'de'],         // List of supported locales in this module
      (locale) => import(\`./generated/locales/\${locale}.js\`) // Dynamic import returning Promise
    );
  }
  
  render() {
    // Usage with standard lit/localize msg function
    return html\`<button>\${msg('Sign In')}</button>\`;
  }
}
`} language="typescript" />

When the global locale changes, the Shell automatically calls the loader function for the new locale and merges the templates.

## 3. Server Overrides

You can override any text in the application (shell or registered modules) by serving external translation files. 
This is useful for adding new translations , or fixing typos or changing terminology without a new release of microfrontend modules.

### Configuration
Configure the shell to look for overrides:

<Source code={`
<polyfea-md-shell
  locales-path="/assets/i18n" 
  locales='["en", "sk", "cz"]'>
</polyfea-md-shell>
`} language="html" />

You can also enable language selection menu in shell actions by setting the `locale-menu` attribute:

<Source code={`
<polyfea-md-shell
    locales-path="/assets/i18n"
    locales='["en", "sk", "cz"]'
    locale-menu>
</polyfea-md-shell>
`} language="html" />

### Override File Format
An override file is a standard JS module exporting a `templates` object. The keys are the translation IDs (generated by lit-localize).
It is recommended to use specific message IDs to avoid accidental overwrites when the source text changes.

**File:** `/assets/i18n/sk.js`
<Source code={`
export const templates = {
  // Overriding a specific message ID
  's4513f8b8f36d33': 'Vlastný nadpis aplikácie', 
};
`} language="javascript" />

The Shell loads this file *after* module translations and merges it on top, effectively replacing the specific strings.

To apply translated strings in your components, always use the `msg` function from [`@lit/localize`](https://lit.dev/docs/localization/overview/).

## Summary of Data Flow

1.  **Input**: User selects 'sk' (Slovak).
2.  **Shell**: Dispatches `loading` status.
3.  **Loaders**:
    *   Shell loads internal `sk` core translations.
    *   Shell loads `sk.js` from `locales-path` (Overrides).
    *   Registered modules (e.g., `@my-org/auth`) load their own `sk.js` via their registered loader function.
4.  **Merge**: All templates are merged. `Overrides` > `Module Translations` > `Source`.
5.  **Render**: Shell dispatches `ready` status and Lit elements re-render with new strings.
